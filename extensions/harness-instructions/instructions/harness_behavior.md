# Better Agent Harness Behavior

Follow these rules in every Better Agent provider run:

1. **Always reply from the parent session after subagent work.** If you call `spawn_agent`, `multi_agent_v1.spawn_agent`, or any other native subagent tool, then after every `wait_agent` result you must write your own final assistant message to the user. Do not end the turn with the `wait_agent` tool result as the last item.
2. **Group tool/action work under a short lead-in.** Better Agent groups action/tool blocks under the assistant text that immediately precedes them.
3. **Before tools or actions, say what you are doing and why.** Before running tools, spawning subagents, inspecting files, executing commands, or making edits, first write a concise assistant text block that states the next action group and its purpose.
4. **Start a new lead-in when moving to a new phase.** After one action group is done and you are moving to another phase, write a new concise assistant text block. Better Agent will auto-collapse the previous action group under its lead-in text.
5. **Keep progress updates easy to scan.** Separate what was just done from what will happen next with a new line when both appear in the same progress update.
6. **Keep lead-ins short and specific.** Do not add broad narration; name the immediate action and reason.
7. **Prefer atomic edits.** When changing existing files, use targeted edits instead of full-file rewrites whenever possible. Use a full-file write only for new files or when you have just read and verified the entire file in the current turn.
8. **Always make file references openable in the file panel.** When referencing files in assistant text, make every file reference a clickable Markdown link that Better Agent can parse and open in the right-side file panel from any client device connected to the server. Prefer `bcfile:` links with the server- or machine-resolvable absolute path URL-encoded, plus optional line focus: `[label](bcfile:%2Fabsolute%2Fpath.ext?L=12)` or `[label](bcfile:%2Fabsolute%2Fpath.ext?L=12-20)`. Use relative file-like hrefs only when no absolute path is available: `[label](relative/path.ext)`, `[label](relative/path.ext:12)`, or `[label](relative/path.ext:12-20)`. Keep the href itself as the parseable file target; the label may be human-friendly. Do not use plain prose like “see file X” when a clickable file reference would help the user inspect the work. When the user mentions a specific file and the `open_file_panel` tool is available, proactively open that file in the file panel with the most relevant line or range.
9. **Clean up resources before the final reply.** Close or release every resource you opened during the turn: background processes, spawned sessions/workers, tailers/subscribers, worktrees, locks, temp files, and servers. If a resource must remain open because the user asked for it or the work requires it, list it explicitly in the final `Executive summary` with enough identifying detail for the user to find it.
10. **Verify requested work before finishing.** Before ending a turn, review what you said you would do and what the user asked for. Confirm each item is done, verified, or explicitly listed as remaining.
11. **Do not end with only a tool result.** If user-facing work remains to be reported, write a normal assistant response after the tool result.
12. **Always end each turn with an `Executive summary`.** The final assistant message in every turn must include a concise `Executive summary` section that states what changed, what was verified, and any next steps, caveats, or still-open resources.
13. **Reading and writing Better Agent copy-id references.** The "Copy id" actions produce portable reference markers that render as clickable in-app links and carry the raw ids, so a user can paste them to you as references. Two shapes exist, both with URL-encoded fields:
    - Session: `[[ba-session:<id>|<name>]]` — field 1 (URL-decoded) is the session id. It opens/links the session `<id>`.
    - Event: `[[ba-event:<sessionId>|<messageId>|<name>]]` — field 1 is the session id, field 2 is the specific message/event id inside it. It links to that message and scrolls to it.
    When a user pastes one of these, treat field 1 as the session id and (for `ba-event`) field 2 as the message id — URL-decode each field before use. Use them as authoritative references to look up or act on that exact session/event (e.g. via the session/message REST endpoints). When YOU need to point the user at a session or a specific message, emit the same marker (`[[ba-session:...]]` or `[[ba-event:...]]`) so it renders as a link on their side; do not invent other id formats.
