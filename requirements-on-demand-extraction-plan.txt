On-demand requirements extraction audit plan

1. Sources of truth
   - Raw provider transcripts stay as durable evidence.
   - requirement_units.jsonl stays as the canonical extracted-requirement log.
   - rg, SQLite FTS, requirements_db.json, and coverage are rebuildable projections.

2. Caller lookup
   - Search extracted units first with both rg and FTS.
   - Search raw native transcript windows only when units are missing, stale, or insufficient.
   - Return only task-relevant processed requirements to the caller.
   - Track every raw window visited.

3. Background extraction
   - After the caller response, run generic requirement-unit extraction on visited raw windows.
   - Append units to requirement_units.jsonl.
   - Update rg/FTS projections and requirements_db.json.
   - Mark windows covered only after units are durably written.

4. Coverage rule
   - Covered means "skip this raw window by default," not "delete evidence."
   - Coverage key: transcript path/sid, element_index window, content fingerprint, extractor version.
   - Reopen when content or extractor version changes, coverage is incomplete, or audit context is needed.

5. Caller response shape
   - Return processed DTOs: text, kind, origin, polarity, strength, source, cwd.
   - Do not expose storage-only fields by default: source_key, unit_index, source_text, files, commits.

6. Boundary
   - Query worker may find evidence and produce extraction candidates.
   - Requirements subsystem owns durable writes and projections.
